# Дизайн: демонстрационное наполнение раздела «База данных»

**Дата:** 2026-03-28  
**Статус:** утверждено в сессии brainstorming (пользователь: вариант A — демо без канона Gen; объём и структура — «ок»)  
**Репозиторий:** `gen-memento-docs`  
**Связанные документы:** `2026-03-28-memento-wiki-design.md` (§6, приложение C), `README.md` (пути `content/db/**`)

## 1. Цель

Наполнить каталог **Базы данных** вымышленными записями, чтобы:

- страницы **`/db/items`**, **`/db/skills`**, **`/db/mods`** и карточки **`/db/.../:id`** выглядели как рабочий каталог, а не как один пример;
- в UI был заметен **фильтр по игре** (несколько значений `game`);
- контент оставался **демонстрационным**: не претендует на канон игры Gen и не требует синхронизации с `gen-sp`.

## 2. Источник данных и пайплайн

- **Формат:** YAML в каталогах, как в текущей реализации `scripts/lib/buildDb.mjs`:
  - `content/db/items/*.yaml` → `type: item`;
  - `content/db/skills/*.yaml` → `type: skill`;
  - `content/db/mods/*.yaml` → `type: modifier`.
- **Схема полей** — как в приложении C родительской спеки: обязательные `id`, `game`, `type`, `name`, `summary`; опционально `body`.
- **Рендер `body`:** на карточке детали по состоянию v1 — **простой текст** с переносами (`pre-wrap`); в спеке контента **не** предполагать MDX, markdown-разметку или виджеты внутри `body` (если позже изменят UI — отдельная спека).
- **Слой `content/db/*/generated/`** для этого объёма **не используется**; только ручные файлы в корне каждой подпапки `items` / `skills` / `mods`.

## 3. Объём и значения `game`

- **Целевой объём:** примерно **5–7 записей** в каждой из трёх категорий (допустимо ±1 при авторинге, если так удобнее сгруппировать смысл).
- **`game`:** ровно два идентификатора для демо:
  - **`gen`** — условная «основная» демо-игра (имя совпадает с slug в экосистеме, **смысл записей вымышленный**, не канон);
  - **`arcade-demo`** — вторая вымышленная игра, чтобы фильтр по игре на списках был полезен.
- **Язык:** `name`, `summary`, `body` — **русский**.

## 4. Идентификаторы и уникальность

- **`id`:** латиница, **kebab-case**, стабильный slug в URL; **уникален в пределах своей папки** (items / skills / mods раздельно).
- **Конфликты:** при дублирующемся `id` в одной категории текущий `buildDb` оставляет последнюю запись при слиянии в объект; при реализации **не допускать** дубликатов — один файл на `id`.
- **Наследие:** файл **`content/db/items/sample.yaml`** (или его содержимое) **включается в общий набор** и **не дублирует** отдельную «демо-запись меча»: либо переименовать/заменить под финальный `id` набора, либо удалить после добавления эквивалентной записи с осмысленным именем файла (`*.yaml` = одна сущность).

## 5. Содержательные ориентиры по категориям

Точные формулировки — на этапе реализации; здесь задаётся **роль** записей, чтобы списки не были однотипными.

### 5.1 Предметы (`items`)

Смесь типов: расходник, экипировка, «квестовый» предмет с шутливым описанием, один **более необычный** артефакт. Часть записей — `game: gen`, часть — `game: arcade-demo`.

### 5.2 Умения (`skills`)

Варианты по смыслу: активное, пассивное, условно «системное»/мета-намёк в духе подсказки — всё в вымышленном тоне. Распределение по двум `game` как у предметов.

### 5.3 Модификаторы (`mods`)

Записи должны читаться как **модификаторы правил или состояния забега** (бафф, дебафф, правило этажа и т.п.) — без привязки к реальным данным Gen. Распределение по двум `game`.

### 5.4 Поле `body`

- У **2–4 записей в каждой категории** добавить **развёрнутый `body`** (несколько предложений или короткий список строк в тексте).
- Остальные могут обходиться одним сильным `summary`.

## 6. Тон и границы

- **Тон:** нейтрально-игровой, допускается лёгкий абсурд; явно ощущается, что это **пример для вики**.
- **Вне объёма:** импорт из `gen-sp`, изменение схемы Zod, MDX внутри карточек БД, i18n, новые поля записи (например теги, редкость) — только отдельной спецификацией.

## 7. Критерии приёмки

- `npm run prebuild` (или эквивалентная сборка с `buildDb`) завершается **без ошибок** валидации YAML.
- В **`db.json`** после сборки три массива **не пусты**; в каждом массиве есть записи с **`gen`** и с **`arcade-demo`**.
- Нет дублирующих **`id`** внутри одной категории.
- Пустой **`sample.yaml`**, висящий отдельно от набора, **отсутствует** (набор согласован с п. 4).

## 8. Следующий шаг процесса

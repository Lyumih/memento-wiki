# Дизайн: модификаторы карт, умений и предметов (Memento)

**Дата:** 2026-04-01  
**Статус:** зафиксировано в сессии brainstorming (вариант **A**: отдельный уровень модификатора; ворота по уровню карты)  
**Репозиторий:** `gen-memento-docs`  
**Связанные документы:** `docs/superpowers/specs/2026-03-30-memento-roll-formulas-design.md` (правило броска `rollCardLevelUp`, ожидания бросков), `docs/IDEAS.md` (п. про модификаторы)

## 1. Цель

Описать систему **встроенных модификаторов** у носителя (карта умения, карта предмета, запись умения в билде — далее **носитель**): когда открываются слоты, как игрок выбирает модификатор, как растёт его сила, как это согласуется с **независимой** прокачкой уровня карты **`L`** и с экономикой бросков Memento.

## 2. Ключевые решения

### 2.1 Независимость прокачки

- **`L`** — уровень **носителя** (карта умения / предмет / и т.д.). Повышается бросками Memento по правилам, зафиксированным в спеке `memento-roll` и коде `rollCardLevelUp`.
- **`Lm`** — уровень **конкретного экземпляра модификатора** в слоте (`Lm ≥ 1`, верхняя граница в продукте согласуется с общей моделью уровней, например до **200**). Повышается **отдельными** бросками Memento по **тому же правилу успеха**, что и для `L` (один закон вероятности на один бросок при текущем уровне).

**Прокачка `L` и прокачка каждого `Lm` независимы:** успехи и неудачи бросков карты не двигают `Lm` и наоборот; отдельные счётчики попыток/событий для модификаторов.

### 2.2 Ворота по уровню карты (слоты)

Возможность **добавить** модификатор в следующий слот открывается только после достижения носителем заданного **`L`**:

- **1-й слот:** `L ≥ 75`
- **2-й слот:** `L ≥ 175`
- **3-й слот:** `L ≥ 275`
- Далее: **`75 + 100·k`** для слота **`k + 1`** (`k = 0, 1, 2, …`), пока система слотов активна в продукте.

Пока `L` ниже порога, слот **не предлагается**; после достижения порога игрок получает **однократное** предложение выбора для этого слота (см. §3). Открытие слота **не** тратит и **не** привязывает бросок карты: это чисто порог по `L`.

### 2.3 Сила модификатора и «удвоение к `Lm = 100`»

- При выборе модификатора задаётся **базовое отображаемое значение** (пример: «двойной удар 40%» на момент выбора при некотором эталонном `Lm`, обычно старт слота).
- **Геймдизайн-ориентир:** к **`Lm = 100`** итоговая величина модификатора ≈ **+100%** к базовому отображаемому значению (пример: 40% → 80%). Точная формула **`value(Lm)`** (линейная, по ступеням, кусочная после 100) — таблица баланса; архитектурно достаточно хранить **`base`**, **`Lm`** и функцию/таблицу роста в определении модификатора.
- Значения **`Lm > 100`** — награда за длительное использование; экстремальные цифры (пример: крит **+90%**) допустимы как редкий потолок при большом ожидаемом числе бросков (порядок **~10⁴** бросков для подъёма с 100 до 200 согласуется с кумулятивом в спеке `memento-roll` для карты — для `Lm` при том же правиле порядок тот же).

## 3. Выбор модификатора (оффер)

- При открытии слота игроку показывается **ровно 3** варианта, выбранных случайно из **пула**, фильтруемого по совместимости (тип умения, оружие, теги урона/школы и т.д.).
- **Дубликаты одного и того же модификатора в тройке разрешены** (осознанное ограничение гибкости). При необходимости баланса в будущем можно добавить правило разнообразия без смены базовой модели.
- Если игроку нужен другой модификатор, чем в оффере: **перекачка носителя** (сброс прогресса карты/умения в смысле дизайна — детали продукта: новый экземпляр, ресет ветки и т.д.). **Реролл оффера за валюту** в v1 не входит; возможен как отдельный баланс-рычаг позже.

## 4. Броски для роста `Lm`

- Один **бросок** повышения `Lm` использует **то же правило**, что `rollCardLevelUp(currentLm, r)` при `r` равномерно из `1…100` (см. спеку 2026-03-30).
- **Когда** выдаётся попытка броска, задаётся **профилем носителя** и типом модификатора: например **использование умения**, **победа**, **поражение** для привязки к контенту; для предметов — свой набор событий (экипировка, получение урона, завершение боя и т.д.). Конкретная матрица триггеров — контент и баланс, не меняющие архитектуру §2.
- Частота попыток (каждое событие = бросок, или накопление «заряда» и один бросок) — **параметр дизайна**; в спеке зафиксировано только: прогресс **`Lm`** и **`L`** разведены, закон одного броска общий.

## 5. Единый каркас для карт, умений и предметов

- Одна модель данных: носитель хранит массив слотов **`[{ modifierDefId, Lm }, …]`** с фиксированным порядком слотов.
- **Карта умения / умение:** пул модификаторов и триггеры из боевой логики.
- **Предмет:** тот же каркас; другой пул и триггеры.
- Текст базового умения может использовать токены `%%` от **`L`** (как в существующем пайплайне); параметры выбранных модификаторов считаются от **`Lm`** (и при необходимости от **`def`**, не подменяя **`L`** без явного правила).

## 6. Применение эффектов и конфликты

- Порядок: **базовое умение** → **модификаторы по слотам** (в порядке индекса слота или явного приоритета в данных) → **внешние** баффы/карты.
- Для цепочек **двойной / тройной удар / крит** в продукте нужна одна явная модель: независимые проверки, взаимоисключающие теги, или приоритет (например тройной перекрывает двойной). Конкретика — в таблице правил боя; здесь зафиксировано требование **отсутствия двусмысленности** для игрока и тестов.

## 7. Таксономия примеров модификаторов (контент)

Группы для фильтрации пула и UI:

| Группа | Примеры |
|--------|---------|
| Цепочка удара | шанс двойного удара, шанс тройного удара, шанс крита |
| Выживание | вампиризм |
| Каденция | снижение перезарядки |
| Ресурс | снижение потребления маны |
| Мощность | «эффективность» — в данных разбить на однозначные типы (`damage_mult`, `heal_mult`, …), избегая общей формулировки без семантики |
| Геометрия и время | площадь %, длительность % |

## 8. Критерии приёмки (для будущей реализации)

- Слот **k** недоступен при `L < 75 + 100k` и становится доступен при достижении порога.
- Рост **`L`** и рост **`Lm`** не смешиваются в одной очереди бросков без явного правила продукта (по умолчанию — полностью раздельные очереди/события).
- Один бросок за **`Lm`** соответствует `rollCardLevelUp` при том же `Lm` как «текущий уровень».
- Оффер из 3 вариантов строится из пула по тегам; выбор сохраняется в слоте.
- Документация игрока и внутренние спеки не противоречат спеке `2026-03-30-memento-roll-formulas-design.md` по вероятностям.

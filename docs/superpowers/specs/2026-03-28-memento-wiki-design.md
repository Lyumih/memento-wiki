# Дизайн: публичная вики Memento Mori

**Дата:** 2026-03-28  
**Статус:** утверждено в сессии brainstorming  
**Репозиторий:** `gen-memento-docs` (Vite + React 19 + Ant Design 6)

## 1. Цель и охват

Сайт-вики описывает систему **Memento Mori** как **общий фреймворк** бесконечного улучшения игровых механик для нескольких игр автора. Игра **Gen** (`gen-sp`) — основной пример на старте, не единственный смысл сайта.

Вики **публичная**. Язык **первой версии: русский** (интерфейс и контент).

Эталонные детали правил для Gen по-прежнему могут жить в `gen-sp` (например, `docs/superpowers/specs/2026-03-28-gen-game-design.md`); вики либо ссылается на них, либо дублирует нормативные формулировки с явной пометкой версии/источника, чтобы виджеты и текст не расходились.

## 2. Аудитория и линии контента

Требование: **смешанная аудитория** — разные разделы под разные цели.

| Линия | Назначение | Тон |
|-------|------------|-----|
| **Для игроков** | Публичное понимание системы | Без внутренних идентификаторов кода; фокус на том, что видно в игре и как ощущается прогресс. |
| **Для разработчиков** | Единая терминология и спека | Точные определения, формулы, соответствие реализациям, границы MVP. |

Статьи помечаются метаданными (frontmatter) и попадают в соответствующее поддерево навигации.

## 3. Информационная архитектура (черновик меню)

1. **Введение / Memento Mori** — что за фреймворк, оси прокачки (смерть / использование / победа), общие принципы ортогональности.
2. **Для игроков** — объяснения, отсылки к разделу «База данных» и к примерам в играх.
3. **Для разработчиков** — нотация `%%`, канонические функции (например, бросок уровня карты), связь с кодом; интерактивные виджеты как «живая спека».
4. **База данных** — каталог сущностей: **предметы**, **умения**, **модификаторы**. У записи обязательна привязка к **игре** (на старте в основном Gen); фильтры или подразделы по типу и игре.
5. **Примеры в играх** — краткие страницы по проектам со ссылками на репозитории/релизы.

**Маршруты:** предсказуемые URL в одном выбранном стиле (например, префиксы `/players/`, `/dev/`, `/db/`, `/games/`). Рендер: **статическая сборка (SSG)** для простого публичного хостинга.

## 4. Контент и смена источника данных

- **Авторинг v1:** файлы в репозитории (Markdown; при необходимости MDX или согласованные расширения для встраивания компонентов).
- **Абстракция источника:** контракт уровня «список страниц + метаданные + тело». Реализация v1 — чтение из файловой системы на этапе сборки; позже возможна подмена на API/CMS **без изменения** контракта маршрутов и типов встраиваемых блоков.

## 5. Интерактивные блоки (виджеты)

- **Формат v1:** готовые **React-виджеты** с полями и слайдерами (уровень карточки, бросок 1–100, параметры токенов `%%` и т.д.), **без** произвольного выполнения кода пользователем в браузере.
- **Логика:** общий модуль с каноническими функциями (например, `rollCardLevelUp` / `resolvePercentToken` по спецификации Memento Mori); виджеты только вызывают этот модуль. Модуль покрывается **unit-тестами** (в т.ч. кейсы из спеки).
- **Встраивание:** именованные блоки в контенте (MDX или зарезервированные fenced-блоки с типом), маппинг на компоненты при сборке.

## 6. Раздел «База данных»

- **Источник v1:** вручную поддерживаемые структурированные записи (YAML/JSON или Markdown с frontmatter) в согласованной структуре каталогов.
- **Задел под импорт:** в пайплайне сборки предусмотреть **точку расширения** (например, prebuild-скрипт, выход в `content/db/generated/` или merge-слой), чтобы позже подмешивать **автовыгрузку из `gen-sp`** (или других игр) без смены URL и шаблонов карточек сущностей.
- **Конфликты имён:** зафиксировать при реализации правило приоритета (например, сгенерированное перезаписывается скриптом; ручные переопределения — отдельный namespace или явный флаг).

**Валидация на сборке:** проверка обязательных полей и типов записей БД; по возможности — предупреждения о битых внутренних ссылках (политика: warn vs fail — выбрать в плане реализации).

## 7. Оболочка UI

Ant Design: layout (боковое или верхнее меню — решение в плане реализации), адаптив, русскоязычный UI.

## 8. Ошибки и тестирование

- Статический сайт: в основном **404**; опционально кастомная страница.
- Тесты: модуль формул (обязательно); при необходимости — дымовые проверки ключевых маршрутов или снапшоты.

## 9. Вне объёма v1 (напоминание)

Редактируемые пользователем code playground с исполнением произвольного TS/JS; полноценный i18n; headless CMS как основной источник (только как будущая реализация `ContentSource`).

## 10. Следующий шаг процесса

После ревью этого файла пользователем — skill **writing-plans**: детальный план реализации (сборка контента, маршрутизация, виджеты, схема БД, CI).

# Дизайн: публичная вики Memento Mori

**Дата:** 2026-03-28  
**Статус:** утверждено в сессии brainstorming  
**Репозиторий:** `gen-memento-docs` (Vite + React 19 + Ant Design 6)

## 1. Цель и охват

Сайт-вики описывает систему **Memento Mori** как **общий фреймворк** бесконечного улучшения игровых механик для нескольких игр автора. Игра **Gen** (`gen-sp`) — основной пример на старте, не единственный смысл сайта.

Вики **публичная**. Язык **первой версии: русский** (интерфейс и контент).

Эталонные детали правил для Gen по-прежнему могут жить в `gen-sp` (например, `docs/superpowers/specs/2026-03-28-gen-game-design.md`); вики либо ссылается на них, либо дублирует нормативные формулировки с явной пометкой версии/источника, чтобы виджеты и текст не расходились.

## 2. Аудитория и линии контента

Требование: **смешанная аудитория** — разные разделы под разные цели.

| Линия | Назначение | Тон |
|-------|------------|-----|
| **Для игроков** | Публичное понимание системы | Без внутренних идентификаторов кода; фокус на том, что видно в игре и как ощущается прогресс. |
| **Для разработчиков** | Единая терминология и спека | Точные определения, формулы, соответствие реализациям, границы MVP. |

Статьи помечаются метаданными (frontmatter) и попадают в соответствующее поддерево навигации. Минимальная схема frontmatter — в приложении A.

## 3. Информационная архитектура и меню v1

Порядок в боковой навигации (см. раздел 7):

1. **Введение / Memento Mori** — что за фреймворк, оси прокачки (смерть / использование / победа), общие принципы ортогональности.
2. **Для игроков** — объяснения, отсылки к разделу «База данных» и к примерам в играх.
3. **Для разработчиков** — нотация `%%`, канонические функции, связь с кодом; интерактивные виджеты как «живая спека».
4. **База данных** — каталог сущностей: **предметы**, **умения**, **модификаторы**. У записи обязательна привязка к **игре** (на старте в основном Gen); подразделы по типу и фильтр по игре в UI.
5. **Примеры в играх** — краткие страницы по проектам со ссылками на репозитории/релизы.

### 3.1 Канонические префиксы URL (v1, стабильный контракт)

Сегменты пути на **латинице** (удобство ссылок и кода); текст интерфейса остаётся на русском.

| Раздел | Префикс | Примечание |
|--------|-----------|------------|
| Введение | `/` | **v1:** одна главная страница (обзор Memento Mori). Дополнительные вводные материалы размещаются как обычные статьи под `/players/` или `/dev/` по аудитории; отдельного префикса `/intro/` в v1 нет. |
| Для игроков | `/players/` | Файлы в `content/players/**` → URL по правилу раздела 3.2. |
| Для разработчиков | `/dev/` | Файлы в `content/dev/**` → URL по правилу раздела 3.2. |
| База данных | `/db/items/`, `/db/skills/`, `/db/mods/` | Списки и карточки сущностей по `id` |
| Примеры в играх | `/games/` | Например `/games/gen` |

Рендер: **статическая сборка (SSG)**. Контракт `ContentSource` обязан отдавать итоговый `path` в точном соответствии с таблицей (чтобы смена источника данных не ломала URL).

### 3.2 Соглашение «файл статьи → URL»

| Источник | Итоговый `path` |
|----------|------------------|
| `content/index.mdx` (или `src/pages` эквивалент, если выбрано в плане) | `/` |
| `content/players/foo/bar.mdx` | `/players/foo/bar` |
| `content/dev/foo.mdx` | `/dev/foo` |
| `content/db/items/{id}.yaml` (или `.mdx` + frontmatter) | `/db/items/{id}` |
| `content/db/skills/{id}.*` | `/db/skills/{id}` |
| `content/db/mods/{id}.*` | `/db/mods/{id}` |
| `content/games/{slug}.mdx` | `/games/{slug}` |

Индексные файлы `index.mdx` в подпапке маппятся на путь каталога без суффикса `/index` (например `content/players/guide/index.mdx` → `/players/guide`).

Список сущностей в каталоге (страница индекса раздела БД) — отдельный маршрут в плане реализации: например `/db/items`, `/db/skills`, `/db/mods` из шаблона или из `content/db/items/index.mdx`.

### 3.3 Значение `audience: both` в навигации

**v1:** у статьи с `audience: both` **один канонический URL**, задаваемый расположением файла (раздел 3.2): файл размещается либо в `content/players/...`, либо в `content/dev/...` — в зависимости от того, под каким префиксом должен открываться прямой линк. В **другом** разделе меню добавляется **второй пункт навигации** (тот же `title`/`order` по согласованию), ведущий на **тот же `path`** (дубликат ссылки, без второго файла и без редиректа).

## 4. Контент и смена источника данных

- **Авторинг v1:** файлы в репозитории. **Формат страниц: MDX** (единый пайплайн для статей с виджетами и без них; обычный markdown внутри MDX допустим). Плагин сборки — совместимый с Vite (например, официальный MDX для Rollup/Vite).
- **Абстракция источника:** контракт уровня «список страниц + метаданные + тело». Реализация v1 — чтение из файловой системы на этапе сборки; позже возможна подмена на API/CMS **без изменения** контракта маршрутов и типов встраиваемых блоков.

## 5. Интерактивные блоки (виджеты)

- **Формат v1:** готовые **React-виджеты** с полями и слайдерами (уровень карточки, бросок 1–100, параметры токенов `%%` и т.д.), **без** произвольного выполнения кода пользователем в браузере.
- **Встраивание v1:** только через **MDX** — импорт именованных компонентов виджетов в теле статьи (например, `<RollLevelDemo />`). Отдельный синтаксис fenced-блоков в v1 **не** вводить, чтобы не плодить парсеры.
- **Логика и норматив:** поведение виджетов и тестов должно совпадать с игровой спецификацией Gen по Memento Mori: репозиторий `gen-sp`, файл `docs/superpowers/specs/2026-03-28-gen-game-design.md` (разделы про `%%`, уровень карточки, бросок, модификации — по мере переноса в общий фреймворк). В репозитории вики — **реализация** минимального набора функций (см. приложение B); при расхождении приоритет у спеки в `gen-sp` до выделения отдельного нормативного документа только для вики.
- **Модуль формул** покрывается **unit-тестами** (кейсы из указанной спеки и граничные значения). Минимальный набор функций — приложение B.

## 6. Раздел «База данных»

- **Источник v1:** вручную поддерживаемые структурированные записи (YAML/JSON или Markdown с frontmatter) в согласованной структуре каталогов.
- **Задел под импорт:** в пайплайне сборки предусмотреть **точку расширения** (например, prebuild-скрипт, выход в `content/db/generated/` или merge-слой), чтобы позже подмешивать **автовыгрузку из `gen-sp`** (или других игр) без смены URL и шаблонов карточек сущностей.
- **Конфликты имён:** зафиксировать при реализации правило приоритета (например, сгенерированное перезаписывается скриптом; ручные переопределения — отдельный namespace или явный флаг).

**Валидация на сборке:** проверка обязательных полей и типов записей БД по схеме из приложения C; проверка внутренних ссылок — **v1: только предупреждение в лог** (сборка не падает).

## 7. Оболочка UI

**v1:** Ant Design **Layout** с **боковым меню** (навигация по разделам) и верхней шапкой (название сайта, при необходимости поиск позже). Адаптив: на узких экранах боковое меню сворачивается (drawer). Русскоязычный UI.

## 8. Ошибки и тестирование

- Статический сайт: в основном **404**; опционально кастомная страница.
- Тесты: модуль формул (обязательно); при необходимости — дымовые проверки ключевых маршрутов или снапшоты.

## 9. Вне объёма v1 (напоминание)

Редактируемые пользователем code playground с исполнением произвольного TS/JS; полноценный i18n; headless CMS как основной источник (только как будущая реализация `ContentSource`).

## 10. Следующий шаг процесса

После ревью этого файла пользователем — skill **writing-plans**: детальный план реализации (сборка контента, маршрутизация, виджеты, схема БД, CI).

## Приложения

### A. Минимальная схема frontmatter статей

| Поле | Обязательность | Описание |
|------|----------------|----------|
| `title` | да | Заголовок страницы. |
| `audience` | да | `players` \| `dev` \| `both` — влияет на размещение в дереве и, при необходимости, на бейдж в UI. |
| `order` | нет | Число для сортировки среди соседей в меню. |
| `game` | нет | Если статья привязана к одной игре (например `gen`). |

### B. Минимальный набор функций модуля формул (v1)

Имена ориентированы на совпадение с кодом `gen-sp` (`src/game/memento/`):

| Функция | Назначение |
|---------|------------|
| `rollCardLevelUp(currentLevel, randomInt1to100)` | Успех +1 к глобальному уровню карточки при использовании (норматив: раздел 4.3 в `2026-03-28-gen-game-design.md`). |
| `rollMementoLevelUp(currentLevel, randomInt1to100)` | **Идентична** `rollCardLevelUp` по сигнатуре и логике; отдельное имя для предметов и прочих сущностей в тексте вики (реэкспорт в коде). |
| `resolvePercentToken` | **Каноническое имя в репозитории вики** для расчёта токенов `%%` (раздел 3 в `2026-03-28-gen-game-design.md`). Имя совпадает с модулем `gen-sp` (`resolvePercentToken.ts`). |

Расширение набора — по мере появления новых тем в контенте вики.

### C. Минимальные поля записи каталога «База данных»

Каждая сущность (независимо от формата файла) должна нормализоваться к виду:

| Поле | Обязательность | Описание |
|------|----------------|----------|
| `id` | да | Стабильный slug в URL (латиница, kebab-case). |
| `game` | да | Идентификатор игры (например `gen`). |
| `type` | да | Одно из: `item` \| `skill` \| `modifier`. |
| `name` | да | Отображаемое имя (русский текст). |
| `summary` | да | Краткое описание для списков. |
| `body` | нет | Развёрнутое описание (markdown/MDX внутри шаблона карточки). |

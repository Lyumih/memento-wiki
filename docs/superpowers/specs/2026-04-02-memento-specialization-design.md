**Дата:** 2026-04-02  
**Статус:** утверждено в сессии brainstorming (вариант **C**: нормативная спека + страница вики + задел под демо-лабораторию)  
**Репозиторий:** `gen-memento-docs`  
**Связанные документы:** `docs/superpowers/specs/2026-03-30-memento-roll-formulas-design.md` (правило одного броска и `P(L)`), `docs/superpowers/specs/2026-04-01-memento-modifiers-design.md` (слоты, оффер из трёх, `Lm`), `docs/IDEAS.md` (п. про специализацию), `content/dev/memento-specialization.mdx`

## 1. Цель

Зафиксировать **специализацию** (условно **«класс Memento»**) как выбор, который **меняет параметры** разрешения Memento для **носителя** (карта умения, карта предмета и т.п. — как в спеке модификаторов), **не отменяя** базовые законы «один бросок — одно правило» из спеки `memento-roll`, а задавая **эффективные** числовые параметры поверх эталона.

Спека описывает **модель и контракт** для эталонного кода в этом репозитории и для переноса в **gen-sp**; конкретные названия классов и баланс пресетов — контент, не часть v1 документа.

## 2. База и оверлей

- **Базовые правила** при отсутствии специализации совпадают с:
  - спекой **memento-roll** (`rollCardLevelUp`, ожидания `E₁(L)`);
  - спекой **memento-modifiers** (пороги слотов `75 + 100k`, оффер из **3** вариантов, независимость `L` и `Lm`).
- **Специализация** задаёт **необязательный** набор **сдвигов** относительно базы. Архитектурно достаточно **чистой функции** вида **`resolveMementoParams(base, specialization) → effective`** (имена модулей в реализации могут отличаться).

Пока у носителя нет специализации или задан **нулевой** пресет, **эффективные** параметры **совпадают** с базой (критерий регрессии для кода и демо).

## 3. Параметры v1 (таблица)

Каждая строка — **опциональное** поле пресета. Значение по умолчанию = база из спек `memento-roll` / `memento-modifiers`.

| Поле | Смысл | База | Пример специализации |
|------|--------|------|----------------------|
| **`levelUpBonusSteps`** | Уточнение ниже, §3.1 | `0` | `1` (условный «+1 к шансу» в терминах исходов `r`) |
| **`modifierOfferCount`** | Число вариантов в оффере при открытии слота | `3` | `4` или `5` |
| **`firstModifierSlotLevel`** | Минимальный **`L`** для **первого** слота модификатора | `75` | `25` |
| **`modifierSlotStep`** | Шаг **`L`** между **соседними** порогами слотов после первого | `100` | `100` (без изменения) или иное целое ≥ `1`, если продукт вводит свой шаг |
| **`previewNextModifierOffer`** | Видимость следующего оффера до фиксации выбора | `false` | см. §3.2 |

Формула порога для слота с индексом **`k`** (ноль — первый слот):  
**`firstModifierSlotLevel + modifierSlotStep · k`**.  
При **`modifierSlotStep = 100`** и **`firstModifierSlotLevel = 75`** получается база **`75 + 100k`**, согласованная с `memento-modifiers` §2.2.

### 3.1 Бонус к успеху броска `L` и `Lm` (`levelUpBonusSteps`)

Правило **одного броска** остаётся тем же, что **`rollCardLevelUp(currentLevel, r)`** в коде, но для носителя со специализацией вводится целое **`b = levelUpBonusSteps ≥ 0`**.

- **Если `currentLevel ≤ 100`:** определить **`L_eff = max(1, currentLevel − b)`**. Успех того же броска, что и в базе, считается по **тому же логическому условию**, что и **`rollCardLevelUp(L_eff, r)`** при **неизменном** `r` из `1…100`.  
  Эквивалентная интерпретация: при **`b = 1`** к множеству успешных значений **`r`** добавляется ровно **один** ранее неуспешный исход (для типичных уровней **`1 < currentLevel ≤ 100`**), без выхода за пределы `1…100`.
- **Если `currentLevel > 100`:** в **v1** поле **`levelUpBonusSteps` не действует** (успех только при **`r === 100`**, как в базе). Расширение для **`L > 100`** — вне объёма v1.

Тот же **`b`** применяется **и** к броскам повышения **`Lm`**, если продукт не вводит отдельного поля; при появлении **`modifierLevelUpBonusSteps`** в данных — оно **перекрывает** **`b`** только для очереди **`Lm`** (зарезервировать в типах, в v1 можно не реализовывать).

### 3.2 Предпросмотр оффера (`previewNextModifierOffer`)

- **`false` (база):** игрок не видит состав следующего оффера до момента, зафиксированного продуктом (как в v1 спеки модификаторов).
- **`true` (игровая механика):** игрок **заранее** видит набор модификаторов, которые будут предложены при **следующем** открытии слота (или при следующем «тикании» оффера — точный триггер задаётся продуктом). **Сильный баланс-рычаг:** в спеке зафиксировано только значение флага; ограничения по рероллам, затратам и мета-слою не входят в v1.
- **Лаборатория на вики (фаза 3):** допускается режим «показать следующий оффер» **только в демо** при **`previewNextModifierOffer === false`** в игровой модели — как инструмент разработчика; это **не меняет** базовую семантику поля для продукта.

## 4. Область действия и стекинг

- **v1:** не более **одной** активной специализации на **носителя**.
- Сочетание специализации носителя с **другими** источниками изменения параметров Memento (мета-предметы, баффы забега) — **вне объёма v1**. При появлении нескольких модификаторов правил приоритет задаётся **отдельной таблицей продукта**; до её появления в дизайне реализация **не обязана** поддерживать стекинг.

## 5. Согласованность с существующими спеками

- Спека **`memento-modifiers`** остаётся верной для носителя **без** специализации: **3** варианта, пороги **`75 + 100k`**.
- Для носителя **со** специализацией число вариантов и пороги берутся из **эффективных** параметров §3; оффер по-прежнему строится из **пула с фильтрацией**, как в базовой спеке, с заменой только **числа выборок** и **независимости** от изменения порогов по **`L`**.
- Вероятность успеха броска **`L`**/`Lm` при **`b > 0`** и **`currentLevel ≤ 100`** **не** описывается отдельной формулой в спеке `memento-roll`: источник истины — **эквивалентность** **`rollCardLevelUp(L_eff, r)`** из §3.1. Страница **`memento-roll`** и виджеты могут позже принимать **опциональный** `b` для сравнения кривых — по плану реализации.

## 6. Примеры архетипов (иллюстрация, не каталог)

| Имя (рабочее) | `levelUpBonusSteps` | `modifierOfferCount` | `firstModifierSlotLevel` | `previewNextModifierOffer` |
|---------------|---------------------|----------------------|---------------------------|----------------------------|
| База | `0` | `3` | `75` | `false` |
| «Широкий выбор» | `0` | `5` | `75` | `false` |

## 7. Задел под демо: `SpecializationLab` (фаза 3)

**Цель виджета:** сравнить **базу** и **пресеты** специализации без игрового UI gen-sp.

**Минимальный контракт (v1 спеки):**

- **Входы:** текущий **`L`**, опционально **`currentLevel`** для расчёта **`P(успех)`** с учётом **`b`**, выбранный **пресет** (набор полей §3).
- **Выходы (отображение):** эффективные **`firstModifierSlotLevel`**, **`modifierSlotStep`**, **`modifierOfferCount`**; при **`currentLevel ≤ 100`** — **`L_eff`** и кратко **число успешных исходов `r`** из `100` (или **`P(успех)`**), согласованное с §3.1.
- **Опционально:** кнопка «следующий оффер» в режиме лаборатории (фиксированный или вводимый seed RNG), **не** смешиваясь с семантикой игрового **`previewNextModifierOffer`** без явной подписи в UI.

Реализация — в плане `docs/superpowers/plans/` после утверждения этой спеки; перенос в **gen-sp** — отдельно.

## 8. Критерии приёмки

- При нулевом пресете эффективные параметры и поведение броска **совпадают** с текущими эталонными функциями вики (`rollCardLevelUp`, пороги слотов, оффер из трёх).
- Любой ненулевой пресет в коде/тестах может быть **сверен** с таблицей §3 и §3.1 без двусмысленности.
- Документация не противоречит спекам **`memento-roll`** и **`memento-modifiers`** там, где действует база.

## 9. Вне объёма v1

- Полный каталог классов, лор, боевые эффекты **вне** изменения параметров Memento.
- Персистентность, экран выбора специализации в **gen-sp**, экономика рероллов оффера.
- Отдельное поле **`modifierLevelUpBonusSteps`** до появления дизайна раздельной прокачки шанса **`L`** и **`Lm`**.

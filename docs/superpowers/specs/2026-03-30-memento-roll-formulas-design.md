# Дизайн: формулы, эмуляция уровня и график на странице `memento-roll`

**Дата:** 2026-03-30  
**Статус:** утверждено в сессии brainstorming (область A; график A; таблицы вех; старт для таблицы — **выбираемый**)  
**Репозиторий:** `gen-memento-docs`  
**Связанные документы:** `content/dev/memento-roll.mdx`, `src/memento/rollCardLevelUp.ts`, `src/memento/resolvePercentToken.ts`

## 1. Цель

На странице разработчика **`content/dev/memento-roll.mdx`** (и в связанных виджетах/модулях):

1. Добавить **явные формулы** расчётов: бросок повышения уровня карты (Memento) и токены `%%` (как в коде `rollCardLevelUp`, `resolvePercentToken`).
2. Добавить **эмуляцию роста уровня**: кнопки «Поднять уровень» (один случайный бросок `r ∈ {1,…,100}` по правилу Memento; при успехе `L := L + 1`) и «Обновить уровень».
3. Сохранить связку **умение / уровень карты** и **токен**: значение токена считается от **текущего уровня эмуляции** `L` (единый источник правды для демо на странице).
4. Показать **график** (вариант **A** из brainstorming): по оси X уровень **`L = 1…200`**, по оси Y **математическое ожидание** числа бросков до **одного** успешного повышения: `E₁(L) = 1 / P(успех | L)` при том же правиле, что в `rollCardLevelUp`.
5. Вывести **ожидаемое число попыток (бросков)** для достижения целевых уровней от **выбираемого старта** `S`:
   - вехи **`10, 20, …, 90, 100`**;
   - вехи **`100, 200, …, 1000`**.  
   Единица — **броски**, не реальное время.

## 2. Математика и соответствие коду

### 2.1 Успех одного броска (`rollCardLevelUp`)

Для текущего уровня карты **`L`** (целое, `L ≥ 1` в демо; код допускает и `0` — в спеке страницы зафиксировать, что демо начинается с **`L ≥ 1`** согласно gen-game-design §4.3, либо явно описать ноль как отдельный случай, если виджет позволит):

- Если **`L > 100`**: успех тогда и только тогда, когда **`r === 100`** → **`P(L) = 1/100`**.
- Если **`L ≤ 100`**: успех, если **`r === 100` или `r ≥ L`**, что для равномерного `r` на `1…100` эквивалентно **`r ∈ {L, L+1, …, 100}`** → число исходов **`101 − L`**, **`P(L) = (101 − L) / 100`**.

**Ожидание числа бросков до первого успеха** (геометрическое распределение): **`E₁(L) = 1 / P(L)`**.

### 2.2 Кумулятив до целевого уровня

Пусть **`S`** — стартовый уровень (целое, **`S ≥ 1`**), **`T`** — целевой уровень.

- Если **`T ≤ S`**: ожидаемое число бросков **`0`** (уже достигли или цель не выше старта).
- Если **`T > S`**: нужно ровно **`T − S`** успешных повышений подряд, уровни по пути **`S, S+1, …, T−1`**. В модели независимых бросков с фиксированным правилом:

  **`E_cum(S, T) = Σ_{L=S}^{T−1} E₁(L)`** (вещественное; не округлять при промежуточных суммах в коде, кроме отображения).

Именно **`E_cum(S, T)`** выводится в таблицах для вех **`T`** при заданном пользователем **`S`**.

### 2.3 Токены `%%`

Формулы и случаи — как в **`resolvePercentToken` / `resolvePercentValue`** (plain / cap / neg, порог **`L > 100`** для заморозки в вариантах cap/neg). В тексте страницы — краткое соответствие спеке §3 / приложению B (без дублирования всей спеки, со ссылкой на код как на эталон поведения).

## 3. Архитектура реализации (рекомендация brainstorming)

- **Чистый модуль** (например `src/memento/cardLevelUpStats.ts` или рядом с `rollCardLevelUp.ts`):  
  `successProbability(L)`, `expectedRollsForOneSuccess(L)`, `expectedRollsToReachLevel(S, T)` — **без UI**, с **юнит-тестами** (несколько уровней: `1`, `50`, `100`, `101`, `200`; кумулятив сверить с ручной суммой для малых диапазонов).
- **График:** без новой тяжёлой зависимости — **SVG** или простая полилиния по точкам `L = 1…200` (Ant Design уже в проекте).
- **Формулы в MDX:** основной текст — **понятная математическая запись** (Unicode / дроби в тексте); при необходимости небольшой React-фрагмент для выравнивания блоков (без обязательного KaTeX в v1).

## 4. Поведение UI на странице

### 4.1 Общее состояние

- **`S`** — **выбираемый старт** (слайдер и/или числовой ввод), диапазон **`S ∈ {1, …, 999}`** (чтобы при необходимости иметь хотя бы один шаг успеха до **`T = 1000`**). Значение **`S = 1000`** не требуется: при **`S ≥ 1000`** все вехи дают **`0`** бросков — при желании UI может ограничивать слайдер **`999`** без отдельной спеки.
- **`L`** — **текущий уровень эмуляции** (карта после бросков).
- **«Обновить уровень»:** **`L := S`** (согласованность с таблицей «достичь от старта **`S`**»).
- **«Поднять уровень»:** сгенерировать **`r`** равномерно из **`1…100`**, вызвать ту же логику, что **`rollCardLevelUp(L, r)`**; при успехе **`L := L + 1`**. Показывать последний **`r`** и факт успех/неуспех (кратко).

### 4.2 Токен

- Строка токена и отображаемое **значение от текущего `L`** — как в текущем **`PercentTokenDemo`**, но **`L`** берётся из общего состояния страницы (не независимый слайдер уровня только для токена).

### 4.3 Таблицы вех

При заданном **`S`**:

- Первая таблица: **`T ∈ {10,20,…,90,100}`** — колонки **`T`**, **`E_cum(S, T)`**.
- Вторая таблица: **`T ∈ {100,200,…,1000}`** — те же колонки. **Веха `T = 100` намеренно присутствует в обеих таблицах** (одинаковое значение **`E_cum(S, 100)`** — дубликат строки допустим для читаемости двух наборов вех).

Если **`T ≤ S`**: везде показывать **`0`** и краткую подпись в UI или вводный абзац («цель не выше старта / уже достигнута»).

**Точность и формат:** **`E₁(L)`** и **`E_cum`** — **вещественные** (сумма обратных вероятностей); в коде считать в **`number`** с обычной осторожностью к большим значениям. В UI — **округление для отображения** (например 1–2 знака после запятой) и/или **разделители разрядов**; при очень больших **`E_cum`** допустима **научная запись** только для **представления**, не для изменения формулы.

### 4.4 График

- **`L = 1…200`** по X, **`E₁(L)`** по Y; подписи осей; краткая легенда, что это **ожидание до одного успеха**, не кумулятив.

### 4.5 Старый ручной режим (опционально)

Текущий **`RollLevelDemo`** со слайдерами «уровень» и «r» можно **убрать**, **свернуть** в «расширенная проверка» или **оставить ниже** — в плане реализации выбрать один вариант с минимальным дублированием; предпочтение: **один основной блок эмуляции + формулы + график + таблицы**, без второго независимого слайдера уровня.

## 5. Файлы в объёме

- **`content/dev/memento-roll.mdx`** — текст формул, порядок секций (формулы → интерактив → график → таблицы или логичная перестановка по согласованию в плане).
- **`src/mdx/MdxShell.tsx`** — регистрация нового составного виджета при необходимости (например **`MementoRollLab`**), чтобы не раздувать MDX импортами.
- **`src/widgets/*`** — реализация UI (или один контейнер + мелкие подблоки).
- **`src/memento/*`** — статистика + тесты.

**Вне объёма:** другие MDX-страницы, страницы БД, смена правил игры в `gen-sp`, KaTeX как обязательное требование v1.

## 6. Критерии приёмки

- Формулы на странице **согласованы** с **`rollCardLevelUp`** и **`resolvePercentToken`** (включая **`L > 100`**).
- Эмуляция: кнопки работают, «Обновить уровень» выставляет **`L = S`**, повышение использует то же правило успеха, что код.
- Токен показывает значение для **текущего `L`** эмуляции.
- График **`E₁(L)`** для **`L = 1…200`** совпадает с модулем статистики (один источник данных).
- Таблицы показывают **`E_cum(S, T)`** для перечисленных вех и реагируют на смену **`S`**.
- **`npm run test`** проходит; при необходимости точечный lint для затронутых файлов.

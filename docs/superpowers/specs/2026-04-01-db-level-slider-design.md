# Дизайн: уровень и числовые параметры на карточках каталога «База данных»

**Дата:** 2026-04-01  
**Статус:** утверждено в сессии brainstorming (согласованы разделы 1–5)  
**Репозиторий:** `gen-memento-docs`  
**Связанные документы:** `2026-03-28-memento-wiki-design.md` (каталог БД), `2026-03-28-db-demo-content-design.md` (демо-контент YAML), `scripts/lib/buildDb.mjs`, `src/pages/DbDetailPage.tsx`

## 1. Цель

На страницах деталей **`/db/items/:id`**, **`/db/skills/:id`**, **`/db/mods/:id`**:

- в **`summary`** и опционально **`body`** показывать **числовые значения**, зависящие от **уровня**;
- дать **один ползунок уровня** на карточке, чтобы при его движении числа в описании **пересчитывались**;
- использовать **единые правила** для предметов, умений и модификаторов (один смысл «уровня» в UI вики).

Контент остаётся **демонстрационным**; спека не привязывает формулы к канону игры Gen.

## 2. Решения по продукту (зафиксировано)

| Тема | Решение |
|------|---------|
| Диапазон уровня | Глобально для всего каталога: **0 … 350**, шаг **1** |
| Модель данных v1 | **A:** структура **`stats`** в YAML + плейсхолдеры **`{{name}}`** в `summary`/`body`; inline-синтаксис в строке **не вводится** в v1 |
| Формула | **`raw(L) = base + perLevel × L`**, где **L** — текущий уровень на ползунке; при **L = 0** значение равно **`base`** |
| Отображение | **`display = Math.floor(raw)`** (дробная часть отбрасывается в сторону −∞; для отрицательных `raw` действует обычный математический `floor`) |
| Дроби в данных | **`base`** и **`perLevel`** могут быть **нецелыми**; целое видит только читатель после `floor` |
| Ползунок без `stats` | Ползунок **всегда виден**; если **`stats` отсутствует или пустой** — слайдер **disabled** и короткое пояснение, что уровень **не меняет** текст |
| Начальное L | При первом открытии карточки **L = 0**; **localStorage** в v1 **не используется** |
| Неиспользуемые ключи `stats` | При сборке: **предупреждение** (warning), если ключ есть в `stats`, но **нигде** не встречается в `summary`/`body` |

## 3. Схема YAML и плейсхолдеры

### 3.1. Поле `stats` (опционально)

На корне сущности, опционально:

```yaml
stats:
  damage:
    base: 10
    perLevel: 2
  duration:
    base: 1.4
    perLevel: 0.3
```

- Ключ показателя — **идентификатор плейсхолдера**; стиль имён: **`camelCase`** на всём репозитории (как в примере).
- У каждого показателя обязательны **`base`** и **`perLevel`**, тип **число** (целое или дробное).

### 3.2. Плейсхолдеры

- В **`summary`** и **`body`** допускаются вставки строго вида **`{{identifier}}`**, где **`identifier`** совпадает с ключом в **`stats`**.
- Регулярное множество имён (v1): **`[a-zA-Z_][a-zA-Z0-9_]*`** (латиница, цифры, подчёркивание; не начинать с цифры).
- Литеральные фигурные скобки в произвольном виде в v1 **не специфицируются**; любая последовательность, совпадающая с шаблоном плейсхолдера, **должна** иметь ключ в **`stats`** (строгий режим).

### 3.3. Ошибки сборки

Сборка **падает**, если:

- у записи есть **`stats`**, но у какого-то показателя нет **`base`** или **`perLevel`**, или они не числа;
- в **`summary`** или **`body`** найден плейсхолдер **`{{name}}`**, для которого **нет** ключа **`name`** в **`stats`** этой же записи;
- в тексте есть **некорректный** фрагмент вида `{{...}}` (пустое имя, недопустимые символы в идентификаторе) — трактовать как ошибку парсинга шаблона.

Запись **без** **`stats`** и **без** плейсхолдеров в тексте — валидна.

## 4. Константы уровня и JSON

- **`LEVEL_MIN = 0`**, **`LEVEL_MAX = 350`** хранятся в **одном месте** в коде фронтенда (например `src/constants/dbLevel.ts`), чтобы UI не расходился со спекой.
- Дублировать эти числа в каждой YAML-записи **не требуется**; при необходимости в `db.json` можно не добавлять поля уровня на сущность.

Расширение типа сущности в TypeScript и содержимое `db.json`: поле **`stats`** как объект с числовыми **`base`** / **`perLevel`** по ключам-показателям (точная форма должна совпадать со схемой Zod в `buildDb.mjs`).

## 5. UI (`DbDetailPage`)

- Блок **«Уровень»**: подпись, отображение текущего **L**, **`Slider`** Ant Design с `min={0}`, `max={350}`, `step={1}`.
- Размещение: логически **с группой контента карточки** (например над **`summary`** или сразу под метаданными) — точный порядок на усмотрение реализации при сохранении визуальной целостности с Ant Design **`Card`**.
- **`summary`** и **`body`**: перед выводом прогнать через подстановку значений при текущем **L**; для **`body`** сохранить **`white-space: pre-wrap`**.
- Состояние **L**: **`useState(0)`** локально на странице; при смене записи (`id`) — **сбрасывать L в 0** (или эквивалентно: состояние привязано к монтированию/ключу маршрута).
- **Доступность:** отображать числовое **L** рядом со слайдером; использовать возможности **`Slider`** по `aria` из Ant Design.

## 6. Логика подстановки (фронтенд)

- Для каждого ключа в **`stats`**: `display = Math.floor(entry.stats[key].base + entry.stats[key].perLevel * L)`.
- Заменить каждое вхождение **`{{key}}`** в строке на строковое представление **`display`** (для целых после `floor` — обычно десятичная запись без дробной части).

Чистые функции **`computeDisplayValue(base, perLevel, L)`** и разбор/замена плейсхолдеров желательно вынести в небольшой модуль для возможных unit-тестов позже.

## 7. Краевые случаи

- **Отрицательный `display`:** в v1 **разрешён**; отдельная валидация «не ниже нуля для всех L» **не входит** в обязательные правила сборки.
- **Очень большие числа:** вывод как есть, без разделителей тысяч в v1.
- **Только `summary` или только `body` с плейсхолдерами:** подстановка независимо по полям.

## 8. Проверка качества

- **Автоматически:** `prebuild` / сборка с обновлённым **`buildDb`** отлавливает ошибки схемы и несоответствия плейсхолдеров; warnings — неиспользуемые ключи **`stats`**.
- **Вручную:** карточка с **`stats`**, карточка без; проверка **disabled**-слайдера, границ **0** и **350**, примера с дробями и ожидаемым **`floor`**.

## 9. Вне scope v1

- Персистентность выбранного уровня (**localStorage**).
- Inline-формулы в тексте (вариант **B** из обсуждения).
- Нелинейные кривые и таблицы **`byLevel`**.
- Локализация формата чисел (пробелы в тысячах и т.д.).
